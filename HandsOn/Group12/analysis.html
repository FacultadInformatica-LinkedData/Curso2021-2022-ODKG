Resource Naming Strategy

The Web relies on the use of Uniform Resource Identifiers as a single global identification system. Defining a consistent, useable, stable and self-describing Resource Naming Strategy, along with following the best practices proposed by standards contribute to enhancing the quality of Linked Data.

Of course, finding the optimal naming policy is not a trivial task and has no one size fits all solution. Thus, sometimes one has to follow their instict onto deciding what might suit best to the problem that their are to solve. We should aim, to our best extent, to try to adhere to the recommendations provided by reputed institutions in the field.

In this work, we propose a draft for the resource naming strategy for the dataset chosen for the practical work, which has been defined attending to the documentation of the subject, and to recommendations by the W3C.

To that end, let us first have a look to the generic syntax of a URI:
URI = scheme:[//authority]path[?query] [#fragment]

Now, let us define the domain on which our data will be located.

SCHEME: Arguably the most popular schemes are http and https. Although there are alternatives like ftp, data, irc. We are sticking with the https scheme, which is the one that fits best our needs.

AUTHORITY: The authority component can be further subdivided into three subcomponents

  • authority = [userinfo@]host[:port]

    • We eschew the use of the userinfo and port subcomponents. The host we propose is "data.eventsatmadrid.org", in which our ontologies will be published. It is worth noting that "eventsatmadrid.org" is an unregistered domain.

    Thus, the domain that we will be using can be formulated as: https://data.eventsatmadrid.org/

    Now that we have defined the domain in which the ontology will be located, let us define the paths and patterns for ontology classes, properties in the ontology and individuals.

  • Definition of ontologies: Because the dataset is rather small and the number of classes and properties that we need to define is not very large, we will be just using one ontology. Thus, ontological terms will be located at https://data.eventsatmadrid.org/ontology If, at any time we redesign the R.N.S. so as to include different ontologies, they will be defined with the following pattern, where [ontologyName] is the name of the ontology. [...]/ontology/[ontologyName]
    • Example of definition of a new ontology: https://data.eventsatmadrid.org/ontology/TouristAttraction
  • Ontological terms are defined using hash URIs (#), verifying the following patterns:
    • Classes:[...]/ontology#[className]
      • Example: https://data.eventsatmadrid.org/ontology#Museum
    • Properties:[...]/ontology#[PropertyName]
      • Example: https://data.eventsatmadrid.org/ontology#Capacity
  • Definition of data: following the pattern [...]/resource/
    • Individuals: defined using slash (/) URIs, which must verify the pattern [...]/resource/[className]/[identifier]
      • Example: https://data.eventsatmadrid.org/resource/Museum/ReinaSofia

    Recap of the naming strategy

    • Domain: https://data.eventsatmadrid.org/
      • Ontology: ontology#
        • Class: [className]
        • Property: [propertyName]
      • Data: resource/
        • Individual: [className]/[identifier]
